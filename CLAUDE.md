# docuseal-java

Java SDK for the DocuSeal API. Hybrid architecture: models are generated from
the OpenAPI spec, the client is written by hand.

## Architecture

- `src/main/java/com/docuseal/models/` - generated by openapi-generator
  (java, library=native, models only), never edit by hand. Regenerate with
  `./generate-types.sh [spec-path-or-url]`. Model names come from
  OpenAPI components: `<OperationId>Request`/`<OperationId>Response` +
  domain-named nested schemas (`TemplateField`).
- `JSON.java`, `RFC3339*.java` - generated supporting files (Jackson mapper
  configuration used by oneOf models and the client).
- `DocusealClient.java` - hand-written client: java.net.http.HttpClient,
  auth header, one method per endpoint mirroring docuseal-js naming
  (`listTemplates`, `createSubmissionFromPdf`, ...). List endpoints take
  hand-written typed params with fluent setters (`ListTemplatesParams`,
  `ListSubmissionsParams`, `ListSubmittersParams`,
  `GetSubmissionDocumentsParams`) - a client concern, like in docuseal-go.
- `CreateSubmissionResponse.java` - hand-written: the response of
  `POST /submissions/init`, which is not in the public spec.
- `generate-types.sh` patches several openapi-generator java bugs
  (see comments in the script): toUrlQueryString/ApiClient coupling, invalid
  generics in oneOf models, `default: false` on the oneOf mask preference.

## Source of truth

The OpenAPI spec is generated by `lib/api_spec/` in the docuseal-website
repository and served at https://console.docuseal.com/openapi.yml. Do not
fix type issues here - fix them in the spec and regenerate.

## Conventions

- `createSubmission` posts to `/submissions/init` (newer response shape),
  matching docuseal-js. The legacy `POST /submissions` array response is not
  exposed.
- `permanentlyDelete*` methods use the DELETE endpoints with
  `?permanently=true`.
- Models use `jakarta.annotation` (`useJakartaEe=true`); deps are Jackson +
  jakarta.annotation-api only.
- Build: `mvn compile`. No tests yet.

## Release

Publishing goes to Maven Central via the Central Publisher Portal
(central.sonatype.com), not legacy OSSRH:

```sh
mvn deploy -Prelease
```

The `release` profile attaches sources and javadoc jars, signs artifacts
with GPG and uploads through `central-publishing-maven-plugin`. Requires:
a portal token in `~/.m2/settings.xml` under `<server><id>central</id>`,
a GPG key, and the `com.docuseal` namespace verified in the portal.
Local check without publishing: `mvn -Prelease package -Dgpg.skip=true`.
